进程监控

解决问题

进程本身是否存活

解决方案

Linux下

  • 传统解决方案

    • ps -aux | grep 进程名
    • 不断轮询，没有输出结果，触发报警
      • 轮询时间间隔问题（难把握）

  • 优雅解决方案

    fork 操作系统源语 父进程执行fork启动子进程 子进程退出会告诉父进程

    • Linux daemontools
      • supervise（二进制工具）

【Supervise监控启动方法】
对RPC、Tomcat等都可以监控

  • supervise二进制
    • 启动需要指定run文件目录
    • run文件存放被监控进程的启动命令
    • 具体启动命令
      • /opt/bin/supervise
      • /opt/conf/run &
    • run文件

      语义监控

      进程是否还工作

解决问题

进程”假死”问题 Zookeeper心跳仅监控心跳，需要模拟请求

解决方案

  • 对服务接口发送请求

    • 模拟用户请求

  • 判断请求返回结果

    • 成功
    • 错误
    • 超时

  • 监控架构层次

    • 进程粒度

  • 监控请求类型

    • HTTP
      • 返回码（200/4**/5**）
    • RPC
      • 独立客户端进程

  • 【问题】

    • 【非幂等请求如何监控】
      • 只监控返回码，不比对结果语义 （PV）
      • 顺序监控请求序列
        • CURD
    • 监控工具
      • 需自研
    • 接口监控任务繁琐

所以不可能覆盖100% 只针对核心接口

错误日志监控

GW Logic DAS 高峰期

解决问题

进程有问题，及时发现 先对通用的解决方案

解决方案

  • 统计线上服务高峰期1分钟错误量
  • 根据高峰期错误量设定报警阈值（分钟）
  • 基于日志实时统计（比如可通过行数）
    • 日志打印本地采集到Hadoop
      • Flume(日志的采集)/Kafka
    • 日志计算分析
      • Spark 【/Storm（弃用）/Flink（研究 统一的批和流）】
    • 计算结果存储
      • Redis/MySQL
    • 定时拉取比对进程
      • 每1分钟拉取结果数据
      • 和设定阈值比较，超过及报警

错误日志报警监控

  • 日志产生
    GW1_error.log按小时切割
    GW2_error.log
  • 采集
    Flume采集 放到Hadoop上的Kafka集群
  • 计算分析
    Spark Streaming 1分钟条数
  • 结果
    存到redis
  • 定时
    每分钟redis拉取结果 超过告警（按照副本的粒度报警）

数据波动监控

第一时间发现流量异常
- 流量异常 竞争对手抓取
- 流量下降 服务入口问题

解决方案

  • 及时流量的对比
    • 天级环比
    • 周级同比
  • 基于实时日志分析
    • 日志采集（Flume） -> 日志缓存（Kafka） -> 日志计算（Spark streaming） -> 结果存储（Redis） -> 结果查询 -> 数据结果同比/环比判定(5s 根据业务) -> 报警（OpenFalcon）

优秀开源监控框架

  • Zabbix（针对机器、数据库比较多）

    • 基于WEB界面的提供分布式系统监视以及网络监视功能的企业级的开源解决方案
    • 监控CPU / 内存 / 磁盘 / 网络 / 端口 / 日志 等功能

  • Open-Falcon

    • 是一款企业级、高可用、可扩展的开源监控解决方案
    • 小米运维团队
    • 支持系统基础监控，第三方服务监控，JVM监控，业务应用监控

  • Prometheus + Grafana

    • Prometheus是一个开源的服务监控系统，它通过HTTP协议从远程的机器上收集数据并存储在本地的时序数据库上。

    • Grafana是一个开箱即用的可视化工具，具有功能齐全的度量仪表盘和图形编辑器，有灵活丰富的图形化选项，可以混合多种风格，支持多个数据源特点。

      K8S Cadvice+hispe 高版本上转到了Prometheus

立体监控解决问题

多维度立体监控

- 多维度、多视角看问题
流量接入 转发 负载 过滤 缓存 重写
业务逻辑 列表 详情 商品 用户 客服
公共逻辑 缓存 推荐 支付 搜索 风控
数据访问 DB 消息 SQL 配置 安全
基础设施 硬件 网络 VM IDC CDN

多视角监控

  • 用户视角 系统可用性、时延、内容完整
  • 系统视角 业务准确性、吞吐能力、系统健壮
  • 旁观者视角 波动、安全、链路、流量路径

层次与维度

流量接入 协议处理，流量处理（负载均衡、清洗、重写…）
业务逻辑 依赖服务的时延，异常比例，某业务流程是否完整
公共逻辑 账户状态监测，支付安全风险，搜索少无策略
数据访问 消息吞吐量，存储慢查询比例，SQL注入风险
基础设施 网卡流量，磁盘IO，CDN回源率，网络丢包率

通用监控项

机器指标 CPU使用率、磁盘使用率、负载、内存使用率、网卡流量
JVM指标 垃圾回收、内存变化、线程状态、YongGCCount、FullGCCount
进程指标 端口存活、进程个数、启动状态、占用FD
DB指标 端口存活、链接可用性、SQL耗时、链接数、锁表、TPS、QPS、数据增量

业务监控项

  • 发布次数
  • 商品信息包含非法字符个数
  • 红包使用次数

立体监控系统特点

  • 业务隔离 原逻辑不变
  • 随时修改 即时生效
  • 运行时不占用资源 零消耗

Agent 聚合方案

方案一 共享内存

无通信协议开销 通过加锁方式来读写 固定内存 C++比较容易实现 对Java不友好
semop(semid,{semnum,-1,SEM_UNDO},1)
semop(semid,{semnum,+1,SEM_UNDO},1)

方案二 TCP

可靠传输 容错方案 存储模型 可能会阻塞业务进程，代价比较高，监控场景不合适
3次Write

方案三 UDP传输

直接扔 对业务流程无阻塞 模型简单
Buffer（Client） -> 收包线程 -> 数据队列 -> 处理线程

策略优化

  • 问题
    UPD：10万QPS（单机）
    共享内存：5000万QPS

  • 解决
    Agent一分钟计算一次
    本地CPU批处理 聚合发送

非阻塞 模型通用 承载能力大幅度提升
